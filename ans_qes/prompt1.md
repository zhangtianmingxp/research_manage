# Prompt 1

## 任务标题

建立研究论文清单、统一数据标准并完成单篇试点

## 本轮定位

这是项目的第一轮正式任务，只完成“清单与数据标准”以及 1 篇论文的结构试点。本轮不批量提取全部论文，不声称完成最终数据汇总，不下载 FASTQ/BAM 等原始大文件，也不生成下一轮 prompt。

执行前必须先阅读并遵守：

- `AGENTS.md`
- `.research_agent/AGENTS.md`
- `PROJECT_RULES.md` 中与工程组织、数据追溯、文档、测试、日志和低上下文模式有关的章节
- `project_plan.md`，重点关注第四至十一节
- `.research_agent/project_state.md`
- `.research_agent/progress.json`

当前基线：

- `文献/研究/` 下应有 29 篇研究论文 PDF，是主清单对象。
- `文献/综述/` 下应有 7 篇综述，只能作为辅助线索，不得进入研究论文主清单或作为最终事实证据。
- 当前没有已完成的 `result`。
- 原始 PDF 必须保持只读。

## 本轮核心目标

1. 为 29 个研究论文 PDF 建立完整文件清单、稳定 ID 和规范书目信息框架。
2. 识别疑似重复 PDF、同一论文的不同版本或预印本/正式发表版本；保留每个原文件，不擅自删除或合并。
3. 固化项目所需的规范化数据表、字段字典、缺失值、受控词表、主外键和唯一键规则。
4. 选择 1 篇证据结构清晰的本地研究论文进行试点提取，用真实证据检验数据模型能否表达实验手段、样本、同步化时间轴、细胞周期阶段、扰动和公共 accession。
5. 输出试点发现的数据模型缺口、证据冲突和后续批量处理前需要人工确认的问题。

## 实施要求

### 1. 低上下文盘点与论文清单

先程序化列出文件名、相对路径、文件大小和基础文件指纹，不直接把 29 篇 PDF 全文同时读入上下文。若需要提取 PDF 文本，优先使用已有轻量工具并按标题页、方法、Data Availability、补充材料引用等目标位置分段读取。

至少建立以下两层记录：

1. `paper_files`：每个本地 PDF 一行，必须完整覆盖 29 个研究 PDF。
2. `papers`：每篇规范论文一行；确认是同一论文的多个文件时可以共享 `paper_id`，但必须保留所有 `file_id` 和关系证据。

ID 要求：

- `file_id` 与 `paper_id` 格式明确、唯一且稳定。
- ID 分配规则写入文档；后续新增论文时不得导致已有 ID 自动重排。
- 不得把未核验 DOI 直接写进 ID。

论文/文件清单至少包含：

- `file_id`、`paper_id`、`relative_path`、`filename`、`file_size_bytes`、文件指纹。
- 原始题名、规范题名、年份、期刊、DOI、PMID（无法确认时使用规定缺失码）。
- `document_role`、`version_relation`、`duplicate_group_id`、`is_canonical_file`。
- 元数据证据定位、提取状态和备注。

重复判定必须使用题名、DOI、作者、年份或正文信息的组合证据。仅文件名相似只能标记为“疑似”，不能直接合并。

### 2. 数据模型与数据字典

按照 `project_plan.md` 建立或定义以下规范化实体：

- `paper_files`
- `papers`
- `experiments`
- `samples_timepoints`
- `perturbations`
- `accessions`
- `evidence`
- `unresolved_issues`
- 最终宽表 `literature_experiment_catalog` 的字段映射规则

每个字段必须在数据字典中说明：

- 字段名、中文含义、数据类型。
- 是否必填、允许的缺失码。
- 受控词表或格式约束。
- 主键、外键和唯一性规则。
- 原始值与标准化值的关系。
- 字段级证据要求。
- 示例值；示例必须明确标记为示意，不得混入真实结果。

至少落实以下缺失码：

- `NR`：论文或补充材料未报告。
- `NA`：不适用。
- `NOT_FOUND`：按规定来源检索后未找到。
- `UNRESOLVED`：来源冲突或无法裁决。
- `RESTRICTED`：受控访问、未公开或受限制。

不得用空字符串掩盖未知信息。对于列表型字段，应规定序列化方式，避免用含义不明的自由文本拼接。

### 3. 必须区分的科学概念

数据模型和试点记录必须明确区分：

- 本文自产数据：`yes`、`no_reused`、`derived_from_own`、`unclear`。
- 实验手段/assay、检测靶点与扰动靶标。
- 同步化操作开始、阻断完成、释放/洗脱及实际采样时间。
- 作者明确声明的周期阶段与推导阶段：`author_stated`、`marker_supported`、`explicitly_inferred`、`unknown`。
- 扰动的预期效果与实际验证结果。
- Project/BioProject、Study/Series、Sample/BioSample、Experiment 和 Run。
- `SRR`、`ERR`、`DRR` 与其他非 Run accession。
- 原始论文样本名、数据库 sample alias 与标准化样本名。

不得仅凭采样时长推断细胞周期阶段；不得仅因 accession 出现在文中就判定其为本文自产数据。

### 4. 试点论文选择

从 29 篇研究论文中选择 1 篇作为试点。选择标准依次为：

1. PDF 文本或表格可可靠定位。
2. 明确报告同步化或细胞周期时间点。
3. 明确包含一种扰动及其对照。
4. Data Availability 中存在可辨认的公共数据编号。
5. 单篇规模适合本轮完成。

先对候选论文做有界检查，再记录最终选择理由。若没有单篇同时满足全部条件，选择覆盖字段最多的一篇，并将缺失维度记为试点发现，禁止为填满表格而猜测。

### 5. 试点提取与证据记录

对试点论文只提取足以验证数据结构的一组代表性实验和相关样本/时间点/扰动/归档记录，不要求在本轮完成该论文所有数据的最终穷举。

试点至少尝试记录：

- 论文 DOI 与书目信息。
- 是否本文自产数据及判定依据。
- 实验手段、检测对象和检测靶点。
- 物种、细胞系/组织/样本名称。
- 同步化技术、试剂/操作、同步化起点、释放与采样时间。
- 采样点对应的细胞周期阶段及证据类型。
- 扰动对象、技术、直接靶标、对照、预期效果和实际验证结果。
- 论文声明的 Project/Study/Series/Sample/Experiment/Run accession。
- 若能由权威官方记录核实，则记录 ENA/EMBL-EBI 页面及 API 返回的文件链接；不得手工拼接链接后标记为已验证。

允许对试点进行有限的官方数据库元数据查询，但只能使用论文、期刊、ENA/EMBL-EBI、NCBI/DDBJ、GEO、BioStudies/ArrayExpress、PRIDE 等权威来源。若执行环境无网络、补充材料缺失或记录受限，使用相应缺失码并写入 `unresolved_issues`；不得因此扩大为全库在线抓取。

每个关键字段必须通过 `evidence_id` 回溯到：

- 本地 PDF 页码/章节/表格/图号，或
- 官方页面/API 查询条件、URL 与查询日期。

证据摘录只保留支持字段所需的最短上下文。来源冲突必须并列保存并进入问题表，不能静默选一个值覆盖另一个。

### 6. 工程实现

优先采用 Python 3.11+ 标准库和少量现有依赖，使用 `pathlib`、类型标注和清晰错误信息。不得为本轮轻量任务随意引入大型依赖。

核心逻辑应放入可复用模块或 CLI，而不是 notebook 或一次性脚本。至少支持：

- 扫描研究/综述 PDF 并生成确定性清单。
- 验证表头、必填字段、缺失码、主外键和唯一键。
- 验证 accession 的实体类型/格式；格式正确不等于在线核验成功，两种状态必须分列。
- 汇总试点记录数量、缺失情况和未解决问题。

所有路径相对仓库根目录或通过参数/配置传入，不硬编码本机绝对路径。长于数秒的操作提供简洁进度或日志；日志不得泄露敏感信息或刷屏。

## 验证与验收标准

执行结束前至少完成以下检查：

1. `文献/研究/` 的 29 个 PDF 在 `paper_files` 中一一出现且无遗漏、无重复路径。
2. 7 篇综述未进入研究论文主表。
3. 所有 ID 唯一；所有外键指向有效记录。
4. 重复/版本关系有证据，疑似关系没有被误标为已确认。
5. 所有规范化表的实际表头与 schema/data dictionary 一致。
6. 必填字段不存在无语义空值；缺失码全部来自受控集合。
7. 试点宽表中的每行能够回溯到规范化记录和至少一条证据。
8. accession 格式校验与在线核验状态严格分开；不得生成虚假的 SRR 或 ENA 下载链接。
9. 运行针对核心逻辑的自动化测试，并记录命令、通过数和失败数。
10. 对工作区做大文件检查，确认未新增原始测序数据、缓存、日志洪泛或 PDF 副本。
11. 确认 `文献/` 下原始 PDF 的路径和内容未被修改。

若当前环境无法可靠核验第 11 项的内容哈希基线，应明确说明只能验证本轮前后路径/当前指纹，而不能声称拥有历史基线。
